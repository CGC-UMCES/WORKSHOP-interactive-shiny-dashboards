Framing

What are data dashboards and when should you build one?

Before we define anything, let's look at a few real examples. [Switch to browser tabs and show 2-3 live dashboards.] Keep it brief — the goal is to give people a feel for the range before we get into definitions.
Section 1 — Framing

What is a data dashboard?

An interactive interface that connects a user to a dataset, letting them explore, filter, and interrogate the data without writing code.

Build a dashboard when…

  • The audience needs to explore, not just read
  • The data updates over time
  • Different users need different views of the same data
  • Spatial or temporal context matters

Prefer a static report when…

  • Findings are fixed and narrative-driven
  • The audience is not expected to interact
  • Long-term archiving is the priority
  • The figure will appear in print
The key word is "without writing code" — the user shouldn't need to be an analyst to get value from it. The left column is when a dashboard earns its complexity. The right column is the honest check — a well-designed static figure and a paragraph of text often does the job just as well, and is much easier to maintain.
Section 1 — Framing

Know your audience before you build

Exploratory
Scientists and analysts who want to dig into the data themselves. Needs flexible filtering, raw data access, and download options.
Operational
Managers or field staff who need current status at a glance. Needs simplicity, clear alerts, and fast load times.
Communicative
Public or policy audiences who need a clear story. Needs guided narrative, minimal controls, and polished visuals.

The same dataset often needs three completely different dashboard designs depending on who is sitting in front of it.

This is probably the most important slide in the whole deck. The same underlying dataset often needs three completely different dashboard designs depending on who sits in front of it. A PI exploring data needs different controls than a program manager checking status, who needs different controls than a public-facing science communication tool. Ask yourself: who is the primary user, and what decision are they trying to make?

Tool Landscape

Choosing the right tool for the job

This section is a quick survey — we are not going deep on any of these tools except Shiny. The goal is to give you enough vocabulary to make an informed choice and to know what your colleagues are talking about when they mention Streamlit or Observable.
Section 2 — Tool Landscape

Overview of dashboard tools

R Shiny
Full reactive web app in R. Excellent for spatial data, custom layouts, and tight integration with R analysis workflows. Requires a server to share.
Plotly / Dash
Python or R. Great for interactive charts and quick deployment. Dash adds a full app framework on top of Plotly.
Google Earth Engine Apps
Purpose-built for satellite imagery and planetary-scale geospatial data. Free hosting, but locked into the GEE ecosystem and JavaScript API.
Observable / Quarto / Shinylive
Serverless options that run entirely in the browser — no server required. Shinylive compiles R to WebAssembly; Observable and Quarto use JavaScript. All make tradeoffs on package support and data size.
See also: Flexdashboard (R)  ·  Streamlit (Python)  ·  Bokeh (Python)  ·  Tableau (GUI)  ·  Power BI (GUI)  ·  Grafana (GUI)
A few things worth flagging: Shiny is now available for Python as well as R — the syntax and concepts are nearly identical. Observable and Quarto are interesting because they produce static HTML files with JavaScript interactivity — no server required, hostable on GitHub Pages for free. Shinylive does something similar but actually compiles R to WebAssembly so real R code runs in the browser. We will come back to all of these in the sharing paradigms section.
Section 2 — Tool Landscape

How to choose

  • Language — use what your team already knows
  • Data size — browser or server?
  • Sharing — permanent URL, demo, or embedded?
  • Interactivity — simple filters or complex reactivity?
  • Maintenance — who owns it after you build it?
Today’s focus: R Shiny
Best fit for research audiences already working in R, complex spatial data, and dashboards that need to stay in sync with live or updating datasets.

Note: Shiny is also available for Python — the syntax and concepts are nearly identical.
The single most important question is language — use what your team already knows. The learning curve for a new framework is real, and the best dashboard is one that gets built and maintained. For research audiences already in R with complex spatial data, Shiny is almost always the right answer. That said, if your collaborators are in Python, Streamlit or Dash might be a better fit for the team even if you personally prefer R.

Section 3 — Sharing Paradigms

Local, static & tunneled

1. Local (localhost)
Default when you click Run App in RStudio. Only accessible on your machine. Zero setup. Good for development and testing.
2. Static HTML (GitHub Pages)
Pre-built HTML file with embedded JavaScript. No server needed. Interactivity limited to reorganizing pre-loaded data — no runtime computation.
3. Tunneled — ngrok
Creates a public URL that forwards to your locally running app. Your computer does all the computation — ngrok just tunnels whatever is running on that port. Free tier gives a temporary URL that changes each session.
# One-time setup
ngrok config add-authtoken TOKEN

# Each session
ngrok http 4533

# Share this URL:
https://abc123.ngrok-free.app

Works for any framework — Shiny, Streamlit, Flask, or anything else running on that port.

Key point about ngrok: your computer is doing ALL the computation. If 20 people open your ngrok URL simultaneously, all 20 sessions are running on your laptop — your CPU, your RAM, your internet connection. It also means if you close your laptop, the app dies. ngrok is framework agnostic — it just tunnels whatever is on that port. You could even run it from a Colab notebook instead of a terminal. Great for demos and quick sharing, not for anything long-term.
Section 3 — Sharing Paradigms

Cloud hosted & serverless

4. Cloud hosted — shinyapps.io
Permanent public URL. Runs independently of your laptop. Free tier: 5 apps, 25 active hours/month. One command to deploy.
5. Serverless — Shinylive, Observable, Quarto
All run entirely in the browser with no server required and can be hosted on GitHub Pages for free. Unlike static HTML, Shinylive can execute real code on demand in the browser. Shared tradeoffs: all data must be bundled with the app, large datasets are not practical, and package support is limited.
# Deploy to shinyapps.io
library(rsconnect)
rsconnect::deployApp()

# Your app gets a URL like:
yourname.shinyapps.io/appname
The key distinction between Static HTML and Serverless: static HTML can only reorganize data that was already baked into the file at build time — no new computation happens. Shinylive actually executes R code on demand in the browser via WebAssembly, so users can trigger real calculations. All three — Shinylive, Observable, and Quarto — can be hosted on GitHub Pages for free, which is a big advantage. The shared tradeoff is that all data must be bundled with the file, so large datasets are not practical.
Section 3 — Sharing Paradigms

Other sharing options

6. Container — Docker
Packages the app and its entire application environment into a container that runs identically anywhere. Solves the “works on my laptop” problem.
7. Institutional server
Many universities run Shiny Server, Posit Connect, or JupyterHub. Common when data cannot leave the institution. Contact your research computing group.
8. Self-hosted
You own the server, the infrastructure, and the maintenance. Maximum control and flexibility. A raw VPS with your own application process and web server.

Once hosted, your app can also be embedded in any webpage via an iframe.

Posit Connect vs Shiny Server: Shiny Server is free and open source but serves Shiny only. Posit Connect is the paid enterprise version that adds authentication, Python support, scheduled reports, and fine-grained access control. Many universities already have one of these — worth asking your IT or research computing group before standing up your own infrastructure. Docker is powerful but requires real DevOps knowledge — the payoff is that your app runs identically anywhere.
Section 4

Large Data Considerations

What happens when your data outgrows the browser

This section is context-setting for the demo app we are about to walk through. The NLCD raster in today's app is deliberately downsampled to ~150m so it fits in the browser — in a production app serving a full-resolution nationwide dataset you would need one of the strategies we are about to cover.
Section 4 — Large Data

Browser rendering limits

  • Browsers render everything as pixels in memory — a 30m nationwide raster is gigabytes before it reaches the screen
  • Leaflet limits raster file sizes for a reason — large images will fail to render or crash the browser
  • Large vector datasets with millions of points or complex polygons also slow the browser
  • Rule of thumb: if the file is over ~5MB, you need a strategy
Strategies
Aggregate on the server before sending to the client

Simplify geometries with st_simplify()

Downsample rasters with aggregate()

Use a tile server or COG
The rule of thumb is ~5MB — above that you need a strategy. This applies to both rasters and vectors. A shapefile of detailed US county boundaries can easily be 50MB+. The browser does not care what the data represents — it has to render every vertex, every pixel. Simplification and aggregation are almost always the right first step before reaching for a tile server.
Section 4 — Large Data

Large Raster Handling

A tile server pre-slices a raster into 256×256px tiles at multiple zoom levels. The browser only fetches tiles for the current viewport — never the whole image at once. This is exactly how Google Maps and satellite basemaps work.
Options in R / open source
COG — Cloud Optimized GeoTIFF. No server needed — the file itself is structured so clients fetch only the bytes they need Zarr — Cloud-native chunked array format ideal for multidimensional data (time × lat × lon). Like COG but for raster time series. Used by ERA5, CMIP6, and other major climate datasets titiler — modern cloud-native tile server GeoServer — full-featured open source tile server gdal2tiles — pre-generate static tile directories
Today’s workshop
We use a coarse-resolution NLCD raster (~150m) that fits within Leaflet’s limits. This keeps setup simple while demonstrating the same raster-vector query concepts you would use with a tile-served layer.
COG is worth highlighting specifically — unlike the other options here, it requires no server infrastructure at all. A COG is just a specially formatted GeoTIFF that supports HTTP range requests, so the browser fetches only the bytes it needs for the current viewport. You can host it on S3 or GitHub and get tile-server-like behavior for free. Zarr is the equivalent for multidimensional data — think daily temperature grids over 40 years. Major climate datasets like ERA5 and CMIP6 are increasingly distributed as Zarr on cloud storage.
Section 4 — Large Data

Large Vector Handling

Simplify before it hits the browser
Reduce vertex count with st_simplify() — usually invisible to the eye, dramatically faster to render.

Aggregate spatially — send county summaries, not parcel-level detail.
Efficient vector formats
FlatGeobuf — like COG for vectors. HTTP range requests fetch only the extent needed. No server, just a static file.

PMTiles — vector tiles in a single file, hostable on S3 or GitHub Pages.

GeoParquet — columnar spatial storage, fast for attribute queries.
FlatGeobuf and PMTiles are the COG equivalents for vector data — both support HTTP range requests and require no server. FlatGeobuf is great for simple feature collections. PMTiles is better when you need tiled vector rendering at multiple zoom levels. The simplification approach with st_simplify() is often the easiest win — a tolerance of 0.001 degrees (~100m) is usually invisible to the eye but can reduce file size by 80% or more.
Section 4 — Large Data

Large Tabular Data

When your dataset is too large to load into memory, or you have millions of rows to render as points, you need tools designed for that scale.
Query instead of load
DuckDB — runs SQL on large Parquet or CSV files directly in your analysis environment. Send only the query result to the browser.

Apache Parquet — columnar format, far faster than CSV for reading subsets. Pairs naturally with DuckDB.
Render at scale
Datashader (HoloViz) — renders millions of points as a rasterized heatmap rather than individual markers. Handles genuinely massive datasets.

DeckGL / Observable Plot — GPU-accelerated rendering of large point clouds directly in the browser.
DuckDB is a game changer for large tabular data — it can query a 10GB Parquet file in seconds without loading it into memory. The pattern is: DuckDB runs the query server-side in R or Python, and only the aggregated result gets sent to the browser. Datashader takes a different approach — instead of reducing the data, it renders all of it as a rasterized heatmap image. The browser receives an image, not millions of individual points, so rendering is instant regardless of dataset size.

R Shiny Syntax

The vocabulary you need before you open RStudio

We are now shifting gears into the vocabulary you will need when we open RStudio. This section is intentionally brief — the goal is to give you just enough to orient yourself in the code, not to make you a Shiny expert. The code walkthrough in Part 2 is where the real learning happens.
Section 5 — R Shiny

Every Shiny app has two parts

UI — the layout
Sets the layout and appearance of the app. Defines input widgets users can interact with and output placeholders the server fills in. Evaluated once at startup — no data, no logic, no computation.
ui <- fluidPage(
  selectInput("variable", ...),
  leafletOutput("map"),
  plotlyOutput("ts_plot")
)
Server — the brains
A function that runs once per user session. Reads inputs, computes results, writes to outputs. Everything inside participates in the reactive graph.
server <- function(input, output, session) {
  output$map <- renderLeaflet({ ... })
  output$ts_plot <- renderPlotly({ ... })
}
The UI is evaluated once at startup — it is just a description of the layout, like an HTML template. No data, no logic. The server is where everything happens — it runs once per user session and participates in the reactive graph. A key insight: data should be loaded OUTSIDE the server function if possible, so it is read into memory once and shared across all sessions rather than re-read on every reactive update.
Section 5 — R Shiny

Inputs and outputs are the wires

Inputs — what users control

# Dropdown
selectInput("variable", ...)

# Date range slider
sliderInput("date_range", ...)

# Checkbox
checkboxInput("show_nlcd", ...)

The server reads these as input$variable, input$date_range, etc.

Outputs — rendered results

In the UI — empty placeholder:
leafletOutput("map")
↓
In the server — fill it in:
output$map <- renderLeaflet({
  leaflet() |> addTiles()
})

The ID "map" in the UI must exactly match output$map in the server.

The inputId and outputId strings are the wiring between UI and server. If they do not match exactly — including case — nothing will render and you will get a silent failure. This is one of the most common beginner mistakes. The arrow between the two code blocks on the right shows this connection — the "map" string in the UI placeholder must match the output$map name in the server exactly.
Section 5 — R Shiny

Reactivity: the engine of Shiny

reactive()
Computes and caches a derived value. Re-runs automatically when its inputs change.
Filters water quality data to the selected parameter and date range
observeEvent()
Triggers a side effect when an event fires. Returns nothing.
When you click a station marker, stores that station as the selected one
render*()
A special reactive wired to a UI output placeholder. Produces visible output.
Draws the time series plot for the selected station and parameter

When an input changes, Shiny automatically re-runs every expression that depends on it — and only those expressions. Dependencies are tracked automatically; you never declare what depends on what.

Reactivity is not unique to Shiny — Observable has a very similar model, Dash has callbacks, and Vue/React in JavaScript are built on the same idea. What makes Shiny distinctive is that you never have to declare what depends on what — Shiny figures out the dependency graph automatically. In Dash, for example, you explicitly declare which inputs and outputs each callback connects. In Shiny you just read input$variable inside a reactive() and Shiny knows to re-run it when that input changes.
Section 5 — R Shiny

Reactivity in code

# reactive() — computes filtered data
wq_filtered <- reactive({
  wq |> filter(parameter == input$variable)
})

# render*() — produces the time series plot
output$ts_plot <- renderPlotly({
  plot_ly(wq_station(), ...)
})
# observeEvent() — responds to map click
observeEvent(input$map_marker_click, {
  rv$selected_station <-
    input$map_marker_click$id
})
The chain in action:
Click a marker → observeEvent stores the station → reactive filters its data → renderPlotly draws the plot
The chain shown here is exactly what happens in mini_app.R when you click a station marker: the observeEvent fires and stores the station ID, which invalidates the reactive() that filters data for that station, which causes renderPlotly() to re-run and draw the updated time series. Three functions, one user action, zero manual wiring. This is the core of what makes Shiny powerful.
